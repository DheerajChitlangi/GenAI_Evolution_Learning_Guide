The Universal Standard for Connecting AI Models to Data Sources

Think "USB for AI" ‚Äî one standardized protocol to connect any AI model to any data source or tool. Created by Anthropic in November 2024 to solve the integration chaos in agentic systems.

üéØ What is MCP?

TL;DR
MCP is an open protocol that standardizes how AI models (Claude, Gemini, GPT-4) connect to data sources (databases, APIs, files, tools). Instead of building custom integrations for each model-tool combination, you build once and it works with all models. Like USB replacing serial ports, parallel ports, etc.

Key Concepts

MCP Client

Lives in your agent/application. Speaks MCP protocol to connect to servers. Makes requests like "read file", "query database", "search emails".

Examples: Claude Desktop, Claude Code, your custom agent

MCP Server

Wraps a data source (database, API, filesystem) and exposes it via MCP protocol. Handles authentication, queries, and returns data in standard format.

Examples: PostgreSQL server, GitHub server, Slack server

MCP Protocol

The standard that both client and server speak. Defines how to: discover available tools, invoke functions, handle errors, stream responses.

Built on: JSON-RPC, similar to LSP (Language Server Protocol)

Installing MCP SDK

# Install MCP SDK (Python) pip install mcp --break-system-packages # Or for Node.js npm install @modelcontextprotocol/sdk

Creating an MCP Server (Python)

from mcp.server import Server, Tool from mcp.server.stdio import stdio_server import psycopg2 # Create MCP server for PostgreSQL server = Server("postgresql-mcp-server") # Define tool that MCP clients can use @server.tool() async def query_database( query: str, params: list = None ) -> dict: """ Execute SQL query on PostgreSQL database. Args: query: SQL SELECT statement params: Optional query parameters Returns: Query results as list of dicts """ # Connect to database (use env vars for credentials) conn = psycopg2.connect( host="localhost", database="banking_platform", user=os.getenv("DB_USER"), password=os.getenv("DB_PASSWORD") ) try: with conn.cursor() as cur: cur.execute(query, params) columns = [desc[0] for desc in cur.description] rows = cur.fetchall() # Format as list of dicts results = [ dict(zip(columns, row)) for row in rows ] return { "success": True, "rows": results, "row_count": len(results) } finally: conn.close() @server.tool() async def get_schema(table_name: str) -> dict: """Get schema information for a table.""" # Implementation... pass # Run server using stdio transport if __name__ == "__main__": stdio_server(server)

Using MCP Server in Your Agent

from mcp.client import ClientSession, StdioServerParameters from mcp import types # Connect to MCP server server_params = StdioServerParameters( command="python", args=["postgresql_mcp_server.py"] ) async with ClientSession(server_params) as session: # Initialize connection await session.initialize() # List available tools tools = await session.list_tools() print(f"Available tools: {[t.name for t in tools]}") # Call tool via MCP result = await session.call_tool( "query_database", arguments={ "query": "SELECT * FROM support_tickets WHERE status = 'open' LIMIT 10" } ) print(f"Found {result['row_count']} open tickets") for ticket in result['rows']: print(f" - {ticket['id']}: {ticket['description']}")

‚öñÔ∏è When to Use MCP?

‚úÖ USE MCP When:

  • Building platform for multiple agents
  • Need governance, security, compliance
  • Working with legacy/hybrid systems
  • Want to swap LLMs without rewrites
  • Reusability matters (3+ agents)
  • Enterprise production systems
  • Multi-team organization
  • Long-term maintainability critical

‚ùå SKIP MCP (for now) When:

  • Single agent, single integration
  • Prototype/POC (speed to market)
  • Ultra-low latency required (<100ms)
  • Simple CRUD with modern REST APIs
  • Won't change/evolve significantly
  • No reusability needs
  • Solo developer/small team
  • Short-lived project

üîí Security & Governance

Key Benefits for Banking

üõ°Ô∏è Centralized Access Control

MCP servers enforce access policies in one place. Don't repeat auth logic in every agent.

üìù Audit Trail

MCP servers log all data access. Single audit trail instead of scattered logs.

üîê Secret Management

Credentials stored in MCP server only. Agents never see database passwords.

üö¶ Rate Limiting

MCP server enforces rate limits. Protect databases from agent runaway queries.

Implementation Example

# MCP server with security enforcement @server.tool() async def query_database(query: str, agent_id: str) -> dict: # 1. Verify agent has permission if not check_permission(agent_id, "database_read"): raise PermissionError(f"Agent {agent_id} not authorized") # 2. Validate query (no DELETE, UPDATE, DROP) if not is_safe_query(query): raise ValueError("Only SELECT queries allowed") # 3. Check rate limit if not rate_limiter.allow(agent_id): raise RateLimitError("Query limit exceeded") # 4. Log audit trail audit_log.record({ "agent_id": agent_id, "action": "database_query", "query": query, "timestamp": datetime.now() }) # 5. Execute query return execute_query(query)

üìä Latency Considerations

Yes, MCP adds a layer. Here's when it matters and when it doesn't:

‚úì Latency Impact: LOW

  • Human-in-the-loop workflows: Support tickets, document review
  • Batch processing: Overnight jobs, data reconciliation
  • Slow backends dominate: Database queries (2-5s), MCP overhead ~50ms (1-2%)
  • Complex data retrieval: API calls, file processing

‚úó Latency Impact: HIGH

  • Real-time chat: Conversational UI with streaming
  • High-frequency operations: Trading decisions, real-time alerts
  • Interactive co-pilots: Code completion, live suggestions
  • Ultra-low latency needs: Requirements under 100ms
üí° Hybrid Approach:
‚Ä¢ Use direct API calls for latency-critical paths
‚Ä¢ Use MCP for complex, governed, reusable integrations
‚Ä¢ Best of both worlds: optimize where it matters
